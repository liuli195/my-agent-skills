# Fix Issues 122 123 124 Design

## Context

本次修复三个已确认的治理问题：

- `cross-agent-review`（跨代理审查）生产入口暴露测试用伪审查结果参数。
- GitHub workflow（工作流）仍含 Node.js 20（运行时）相关旧版本引用。
- `pr-flow diagnose`（拉取请求流程诊断）在缺 upstream（上游分支）时提示手工 push（推送），但 `complete`（完成）已经能安全自动 push（推送）。

## Approach

采用最小删除和局部替换。

### Cross-Agent Review

删除 `--fake-reviewer-results`（伪审查结果）参数、数据字段、解析函数和 dispatch（派发）绕过分支。测试不再走 CLI（命令行接口）伪造结果；需要 reviewer（审查代理）输出时，直接 monkeypatch（测试替换）`dispatch_reviewers`（派发审查代理）。

不保留兼容，因为这个入口可要可不要，且本身是问题根因。

### GitHub Workflows

只更新现有版本引用：

- `actions/checkout@v4` -> `actions/checkout@v5`
- `actions/setup-node@v4` -> `actions/setup-node@v6`
- `node-version: "20"` -> `node-version: "24"`
- `actions/setup-python@v5` -> `actions/setup-python@v6`

`github/codeql-action/*@v4` 保持不变，因为当前没有 v5 主版本可升级。

验证时扫描所有 active workflow（活动工作流）和 release-flow workflow template（发布流程工作流模板）中的 `uses:` 与显式 runtime（运行时）版本。能升级的必须升级；保留的版本必须有明确例外断言。

### PR Flow

`diagnose`（诊断）缺 upstream（上游分支）时不再停在手工 push（推送）建议。输出 `DISPATCH_REQUIRED`（需要外部进展），`reason`（原因）保持 `missing_upstream`（缺少上游分支），details（详情）保留 `branch`（分支）和 `baseBranch`（目标分支）。`nextCommand`（下一步命令）复用现有 `pr_body_next_command("complete", ...)` 生成，必须带 `--summary`（摘要）和 `--scope`（范围）。

`complete`（完成）继续是唯一执行 push（推送）的入口，复用已有工作区干净、非目标分支、远端规则和 push 失败处理。

## Tests

- `tests/test_cross_agent_review_cli.py`
- `tests/test_release_flow_cli.py`
- 现有 package/build validation（包/构建校验）测试中补 workflow（工作流）旧版本扫描
- `tests/test_pr_flow_cli.py`
- `openspec validate fix-issues-122-123-124 --strict`

端到端回归保持本地、无远端副作用：

- cross-agent-review（跨代理审查）通过 `run` CLI（命令行接口）读取真实 `review-input.json`。
- release-flow（发布流程）通过模板/当前仓库 workflow（工作流）测试覆盖生成发布形态。
- PR Flow（拉取请求流程）通过临时本地仓库从用户入口运行 `diagnose`（诊断）缺 upstream（上游分支）路径。

## Risks

- 删除伪审查参数会破坏旧测试调用；测试会同步改成内部 monkeypatch。
- Action（工作流动作）升级对旧 self-hosted runner（自托管运行器）可能有要求；本仓库 workflow 使用 GitHub-hosted `ubuntu-latest`（GitHub 托管 Ubuntu 最新环境），风险低。
