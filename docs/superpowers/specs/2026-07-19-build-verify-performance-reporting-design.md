# Build and Verify（构建与验证）性能预算警告与诊断报告技术设计

## 背景

Build and Verify（构建与验证）已经记录每个检查项的耗时，也能在 `verify --full`（完整验证）中执行全部检查，但目前不能回答“本次完整验证是否超过仓库自己的总耗时目标”，也没有稳定的机器可读诊断报告。

本变更在整个 `plugins/build-and-verify`（构建与验证插件）范围内补齐这一能力，同时保持现有边界：不读取业务测试输出、不修改目标仓库的 CI（持续集成）、不安装依赖、不增加逐项预算，也不让性能波动阻断功能验证。

## 设计目标与约束

- 目标仓库可选配置正整数 `verify.fullBudgetSeconds`（完整验证预算秒数）；插件不提供仓库专用默认值。
- 总耗时只覆盖完整检查的实际调度阶段，不包含参数解析、配置读取和报告写入。
- 所有检查必须完整运行后才判断预算；超预算只产生警告和报告，不改变功能验证退出状态。
- `--performance-report`（性能报告）只允许与 `--full`（完整验证）同时使用。
- 超预算时自动写报告；未超预算或未配置预算时，只在显式请求后写报告。
- 报告固定覆盖 `.build-and-verify/runs/performance-report.json`，不保留历史。
- 只复用 Python（Python 语言）标准库和现有 `CheckResult`（检查结果），不新增模块、依赖或通用观察器抽象。

## 总体流程

一次完整验证按以下顺序执行：

1. 命令入口解析 `--full` 与 `--performance-report`，拒绝“请求性能报告但未启用完整验证”的参数组合。
2. 运行器读取并校验配置；`verify.fullBudgetSeconds` 缺省时表示不判断预算，存在时必须是正整数。
3. 运行器记录单调时钟起点，然后调用现有完整检查调度。
4. 调度器完成所有检查并返回按配置顺序排列的 `CheckResult` 列表，同时保留现有输出、缓存处理和失败汇总行为。
5. 运行器记录总耗时，确认结果完整，并根据检查结果确定功能验证状态。
6. 结果完整时再判断预算；超预算时输出性能警告，并根据触发矩阵决定是否写入报告。
7. 报告操作完成后，返回原功能验证退出状态。

`time.monotonic()`（单调时钟）用于计算持续时间，避免系统时间调整影响耗时；报告生成时再记录一个 UTC（协调世界时）`generatedAt`（生成时间）。

## 运行时改动

### 命令入口

`plugins/build-and-verify/skills/build-and-verify/scripts/build_and_verify.py` 在现有 `verify`（验证）子命令上增加布尔参数 `--performance-report`。入口在调用运行器前执行参数组合校验：若提供该参数但未提供 `--full`，则给出明确错误并以现有参数错误方式退出，任何检查都不运行。

入口把显式报告意图以及现有运行时版本传给 `run_verify`（运行验证）。运行时文件清单仍只有现有两个脚本和版本元数据，不增加第三个性能模块。

### 配置校验

现有配置校验器接受可选的 `verify.fullBudgetSeconds`：

- 字段缺失：预算为 `null`，不作超预算判断。
- 值为正整数：参与完整验证结束后的预算比较。
- 值为布尔值、零、负数、浮点数、字符串或其他类型：在检查调度前报告字段错误并退出。

Python（Python 语言）的 `bool` 是 `int` 的子类型，因此校验必须显式排除布尔值。

### 调度结果传播

`_run_scheduled_checks`（运行调度检查）已经持有 `CheckResult`。本次只扩展其返回值，使调用方取得按配置顺序排列的结果列表；不改变检查启动、并行限制、输出顺序、缓存写入和失败标识的现有含义。调用方用结果数与所选检查数判断本次完整验证是否真正完整；不完整时跳过全部性能逻辑。

`run_verify`（运行验证）只在完整模式中建立总计时边界。快速模式不计算预算、不写性能报告，也不清理固定报告。

### 预算判断与警告

原始总耗时先统一保留两位小数，得到公开的 `totalSeconds`。只有配置了预算时才用该公开值计算 `overBudget`（是否超预算）：

- `totalSeconds > budgetSeconds`：超预算。
- `totalSeconds <= budgetSeconds`：未超预算；恰好等于预算不算超限。
- 未配置预算：`overBudget` 为 `null`。

超预算警告在所有检查结束后输出，固定包含总耗时、预算、超出秒数、超出百分比和报告路径。百分比按 `(totalSeconds - budgetSeconds) / budgetSeconds * 100` 计算。性能警告不替换检查失败信息，也不改变返回值。

## 报告生成

### 触发矩阵

| 是否超预算 | 是否显式请求 | 处理 |
|---|---|---|
| 否或无预算 | 否 | 不生成，也不修改已有固定报告 |
| 否或无预算 | 是 | 生成报告 |
| 是 | 否 | 自动生成报告并输出性能警告 |
| 是 | 是 | 生成报告并输出性能警告 |

固定报告路径相对于目标仓库根目录解析，不相对于插件源码目录或当前 shell（命令行解释器）的偶然工作目录解析。

### 固定数据契约

报告为 UTF-8（统一字符编码）JSON（结构化数据），`schemaVersion` 固定为 `1`：

```json
{
  "schemaVersion": 1,
  "runtimeVersion": "0.1.37",
  "generatedAt": "2026-07-19T12:00:00Z",
  "totalSeconds": 12.35,
  "budgetSeconds": 10,
  "overBudget": true,
  "verificationStatus": "passed",
  "checks": [
    {"id": "tests", "status": "passed", "durationSeconds": 8.10}
  ]
}
```

字段规则如下：

- `verificationStatus`（验证状态）只取 `passed`（通过）或 `failed`（失败），由功能检查结果决定。
- `checks`（检查项）包含所有已配置检查，保持配置顺序。
- 未配置预算时，`budgetSeconds` 与 `overBudget` 都为 `null`。
- 没有配置检查时，`checks` 为空，总耗时仍是有效非负值。
- 总耗时和逐项耗时在公开前统一保留两位小数；预算判断、终端输出和报告使用同一公开总耗时。

### 写入

报告逻辑留在现有运行器文件内，使用标准库完成：

1. 确保 `.build-and-verify/runs` 目录存在。
2. 在同一目录写入固定临时文件。
3. 完成 JSON（结构化数据）序列化与刷新后，用原子替换覆盖最终报告。
4. 写入成功后输出最终报告路径。

未触发报告时不触碰已有固定报告。报告写入或替换失败时，输出 `performance-report-warning`（性能报告警告），但不改变功能验证状态。

固定文件只表达“最后一次完成且需要报告的完整验证”。同一目标仓库并发运行多个完整验证时采用最后完成者覆盖；本变更不承诺并发历史隔离。如未来需要历史或并发追踪，应另行设计按运行标识分文件的能力。

## 退出状态优先级

性能诊断不是新的验证门禁。退出状态保持以下规则：

| 功能检查 | 性能结果 | 报告操作 | 最终结果 |
|---|---|---|---|
| 全部通过 | 预算内或超预算 | 成功或失败 | 成功 |
| 存在失败 | 预算内或超预算 | 成功或失败 | 功能验证失败 |

参数错误和配置错误仍按现有规则在检查前失败。`KeyboardInterrupt`（键盘中断）导致结果数少于所选检查数时跳过性能判断和报告；进程被终止或机器掉电时不保证检查、警告或报告能够完成。

## Init Skill（初始化技能）与文档契约

Build and Verify Init（构建与验证初始化）的现有问卷增加一个可选问题：是否为完整验证配置总耗时预算。只有用户选择启用并确认正整数后，配置草案才写入 `verify.fullBudgetSeconds`。

问卷、草案展示、最终确认和写后校验必须一致说明：

- 预算是完成后的性能警告，不会提前终止或阻断验证。
- 超预算会自动覆盖固定性能报告。
- 不启用时完全省略该字段，不写入通用默认值。

Build and Verify Skill（构建与验证技能）同步说明命令用法、报告触发矩阵、固定路径和退出状态边界。插件 manifest（清单）与运行时版本元数据按仓库既有发布规则同步更新，使安装包声明与实际可复制运行时一致。

## 测试策略

### 进程内回归

优先扩展现有 Build and Verify（构建与验证）测试，使用 fake runner（假执行器）和可控单调时钟覆盖：

- 预算内、等于预算、超预算和未配置预算。
- 显式报告与自动报告的完整触发矩阵。
- 功能成功、功能失败与报告失败之间的退出状态优先级。
- 完整固定 JSON（结构化数据）字段、UTC（协调世界时）生成时间、检查顺序和空检查列表。
- 原子覆盖及写入失败警告。
- 结果不完整时跳过性能逻辑，快速模式不读取预算也不触碰已有报告。
- 非正整数预算和 `--performance-report` 未配合 `--full` 的前置拒绝。

进程内测试不运行目标仓库真实业务测试，避免性能保护机制和本仓库测试实现耦合。

### 发布形态端到端验证

使用一个临时目标仓库，从插件发布入口初始化或复制实际运行时，再通过命令入口执行最小的完整验证：

- 超预算时全部检查完成、自动报告生成且功能成功仍返回成功。
- 未超预算时显式参数生成相同格式报告。

临时检查命令只用于验证通用运行器协议，不依赖本仓库业务测试。验证后清理临时目标仓库。

### 仓库级验证

实现完成后执行 OpenSpec（开放规格）严格校验、Build and Verify（构建与验证）聚焦测试，以及本仓库现有 build（构建检查）、fast verify（快速验证）和 full verify（完整验证）主流程。整个过程不为测试本能力而修改根目录业务配置。

## 发布与回滚

发布沿用插件现有版本和运行时同步流程。目标仓库只有在显式执行 `update-runtime`（更新运行时）后才获得新参数和配置支持，插件不会主动改写目标仓库。

未配置预算且未使用新参数的仓库行为不变。回滚到旧运行时前，应移除新配置字段和命令参数；固定性能报告是诊断产物，可安全忽略或删除，不参与功能验证状态。

## 不采用的方案

- 独立性能报告模块：当前只有一个调用点和一种报告，新增模块会扩大运行时同步面。
- 通用结果观察器：当前没有第二个结果消费者，属于未请求抽象。
- 逐项预算、历史基线、统计采样、自动调优和二分定位：复杂度与本次“简单总耗时警告和固定报告”的目标不匹配。
