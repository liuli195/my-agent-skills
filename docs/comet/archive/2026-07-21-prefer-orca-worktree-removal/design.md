## Context

`remove_worktree()` 目前只调用非强制 `git worktree remove`（Git 工作树删除）。当目标目录仍被 Orca（工作区管理器）终端占用时，Git（版本管理）可能注销工作树登记却无法删除目录，后续无法由原路径重试。Orca（工作区管理器）已维护其工作树和终端的运行态，应优先协调其已登记的工作树。

## Goals / Non-Goals

**Goals:**

- 对非主、干净的目标工作树，按 Orca（工作区管理器）登记状态选择删除器。
- Orca（工作区管理器）已登记时只调用非强制 Orca（工作区管理器）删除，并保持现有 Git（版本管理）回读确认。
- Orca（工作区管理器）未登记、不可执行或返回无效工作树列表时无缝回退现有 Git（版本管理）删除。
- Orca（工作区管理器）已登记但删除失败时保留诊断并停止，避免 Git（版本管理）部分删除。

**Non-Goals:**

- 不启动、配置或安装 Orca（工作区管理器）。
- 不关闭 Orca（工作区管理器）终端，不使用 `--force`（强制）或修改 Orca（工作区管理器）配置。
- 不改变 PR（拉取请求）合并、源分支删除、目标分支同步或 `--remove-worktree`（删除工作树参数）的调用时机。

## Decisions

### 先保留现有安全拒绝边界

继续先读取 Git（版本管理）工作树清单、拒绝主工作树并拒绝脏目标工作树。只有目标已通过这些检查后才执行 Orca（工作区管理器）探测，避免对本不允许删除的路径调用外部运行时。

### 用 Orca（工作区管理器）工作树列表按路径匹配

新增最小封装执行 `worktree ps --json`（工作树列表）并解析 `result.worktrees`。以现有 `normalized_path()`（规范化路径）比较目标路径和 Orca（工作区管理器）条目的路径；匹配时使用返回的完整 `worktreeId`（工作树标识）作为删除选择器。

命令不可执行、退出失败、JSON（数据格式）无效或结构不符合预期时都视为“不可用或未登记”，直接使用现有 Git（版本管理）删除。这样普通 Git（版本管理）工作树和未安装 Orca（工作区管理器）的环境没有新配置或依赖。

### Orca（工作区管理器）匹配后只使用 Orca（工作区管理器）删除

匹配条目时执行 `worktree rm --worktree id:<worktreeId> --json`（Orca 工作树删除），不传 `--force`（强制）。命令成功后仍重新读取 Git（版本管理）工作树清单确认目标已消失，保持既有删除后验证。

若 Orca（工作区管理器）删除失败，抛出 `orca_worktree_remove_failed`（Orca 工作树删除失败）并保留退出码、标准输出和标准错误；不得再调用 `git worktree remove`（Git 工作树删除）。这避免 Orca（工作区管理器）与 Git（版本管理）对同一路径产生部分完成状态。

### 跨平台定位 Orca（工作区管理器）命令

优先使用 `ORCA_CLI_COMMAND`（Orca 命令环境变量）；开发环境使用 `orca-dev`（Orca 开发命令）；Linux（Linux 系统）使用 `orca-ide`（Orca 集成开发环境命令）；其他平台使用 `orca`（Orca 命令）。找不到命令时走 Git（版本管理）回退，不把 Linux（Linux 系统）屏幕阅读器误当作 Orca（工作区管理器）。

## Risks / Trade-offs

- [Orca（工作区管理器）登记数据无法读取] -> 视为不可用并保留已验证的 Git（版本管理）删除行为。
- [Orca（工作区管理器）删除被终端或权限阻止] -> 返回 `orca_worktree_remove_failed`（Orca 工作树删除失败），不尝试第二个删除器。
- [Orca（工作区管理器）条目路径格式不同] -> 用现有规范化路径比较，并为 Windows（视窗）大小写差异增加回归测试。
- [外部命令测试不稳定] -> 通过现有命令桩隔离 Orca（工作区管理器）成功、不可用和失败分支，不依赖真实桌面运行时。

## Migration Plan

无需配置或数据迁移。部署后，带 `--remove-worktree`（删除工作树参数）的既有调用在未受 Orca（工作区管理器）管理时保持原行为；Orca（工作区管理器）登记的目标自动使用其删除器。
