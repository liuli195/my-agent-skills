## Context

This change fixes three governance issues with small, local edits:

- `cross-agent-review run` still exposes the old `--fake-reviewer-results` test shortcut.
- Repository GitHub workflows and the release workflow template still use Node.js 20-era action/runtime references.
- `pr-flow diagnose` stops at a manual push instruction before it can point users to the existing `complete` lifecycle.

The repository already has the needed seams: cross-agent-review tests can monkeypatch in-process functions, release workflow tests read workflow YAML as text, and PR Flow already has `pr_body_next_command` plus safe auto-push inside `complete`.

## Goals / Non-Goals

**Goals:**

- Remove fake reviewer results instead of hiding or compat-wrapping them.
- Upgrade all active repository workflow action/runtime references that have current non-deprecated replacements.
- Keep CodeQL action references on their current major because no newer major exists.
- Make `diagnose` point unpushed feature branches to `complete` while keeping the unpushed fact in stop-state details.

**Non-Goals:**

- No new dependencies.
- No new public flags, compatibility mode, or test-only CLI surface.
- No rework of PR Flow push safety logic.
- No workflow structural changes beyond version values.

## Decisions

1. Delete the fake review path from production code.
   - Alternative: hide it behind an environment variable. Rejected because the user explicitly wants optional paths removed.
   - Tests that need reviewer output should reuse Python monkeypatching around `dispatch_reviewers`, not a user-visible CLI flag.

2. Upgrade only existing workflow version references.
   - `actions/checkout@v4` becomes `actions/checkout@v5`.
   - `actions/setup-node@v4` becomes `actions/setup-node@v6`.
   - `node-version: "20"` becomes `"24"` for the full verification workflow.
   - `actions/setup-python@v5` becomes `actions/setup-python@v6`.
   - `github/codeql-action/*@v4` stays at v4 because there is no v5 line to move to.
   - Verification scans every active `.github/workflows/*.yml` file and the release-flow workflow template for `uses:` and explicit runtime version references. Any reference with an available current major is upgraded; any retained reference is documented by test expectations.

3. Keep push execution owned by `complete`.
   - `diagnose` should explain the branch has no upstream, but its actionable `nextCommand` should be the existing `complete` command.
   - This avoids duplicating remote rules checks and push behavior in a read-only diagnosis path.
   - The missing-upstream diagnose state is `DISPATCH_REQUIRED` with `reason: missing_upstream`, `branch`, `baseBranch`, and a `nextCommand` generated by the existing `pr_body_next_command("complete", ...)` path so it includes `--summary` and `--scope`.

## Risks / Trade-offs

- GitHub action major upgrades can affect old self-hosted runners. This repository uses `ubuntu-latest`, so the risk is low.
- Removing the fake review CLI flag breaks tests that used it directly. The migration is local to tests and removes a production shortcut.
- `diagnose` will no longer emit `PUSH_REQUIRED` for the missing-upstream preflight case. Push failures inside `complete` still emit `PUSH_REQUIRED`.

## Migration Plan

No runtime data migration is needed. Existing users should run `complete` for the new branch lifecycle. Tests should stop invoking `--fake-reviewer-results`.

End-to-end checks stay within local and non-destructive boundaries:

- cross-agent-review is exercised through the `run` CLI with a valid `review-input.json`, with reviewer output supplied by in-process monkeypatching for local tests.
- release-flow workflow generation/validation is exercised through the existing template/current-repo workflow tests, not a live GitHub release.
- pr-flow diagnose is exercised through the CLI on a local temporary repository branch without upstream.
