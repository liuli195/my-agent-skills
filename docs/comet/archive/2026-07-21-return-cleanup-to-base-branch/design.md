## Context

当前 cleanup（清理）在本地目标分支未被其他 worktree（工作树）占用时，先将当前工作树置于远端目标提交的 detached HEAD（分离头），再更新本地目标分支。该流程保护了其他工作树，却使普通单工作树使用者在成功后仍需手动切换分支。

本次只改变 cleanup（清理）成功后的当前工作树位置。远端目标提交仍以本次 `git fetch`（拉取）后的快照为准；不得引入 `git pull`（拉取）或跨工作树写入。

## Goals / Non-Goals

**Goals:**

- 目标分支未被其他工作树占用时，使 cleanup（清理）完成后当前工作树检出该分支，并验证三个提交引用一致。
- 目标分支被其他工作树占用但已同步时，继续安全清理源分支，并报告保留 detached HEAD（分离头）的原因。
- 在任一可能删除源分支的操作前完成目标分支安全性与当前工作树切换可行性检查。

**Non-Goals:**

- 不修改其他工作树、不强制删除工作树，且不引入 `git pull`（拉取）。
- 不改变目标分支落后、源分支被占用或本地独有提交时的拒绝行为。
- 不改变 PR（拉取请求）合并、远端保护规则、Pi（编码助手）入口或 `--remove-worktree`（删除工作树参数）契约。

## Decisions

### 固定远端快照后决定当前工作树位置

cleanup（清理）继续在一次 `fetch`（拉取）后解析远端目标提交，并将其作为本次所有比较与回读的唯一快照。它先切换至该快照的 detached HEAD（分离头）以释放源分支；目标分支未被其他工作树占用时，再安全更新并检出该分支。检出成功且三个引用一致后才删除源分支。

不使用 `git pull`（拉取），因为它会在清理流程中引入额外的远端推进与合并行为，破坏既有快照边界。

### 目标分支被占用时采用已同步降级

当 `git worktree list`（工作树清单）显示其他工作树检出目标分支，且该本地分支已等于远端快照时，当前工作树保持在该快照的 detached HEAD（分离头）。此时不尝试检出或更新被占用的分支，完成源分支清理，并在完成详情写入 `baseCheckout.status: skipped` 和 `baseCheckout.reason: base_branch_occupied`，同时保留占用工作树列表。

目标分支检出失败时，系统 MUST 重新读取工作树清单，而非解析 Git（版本管理）错误文本：若新出现的占用分支已等于远端快照，则沿用该降级路径；否则在删除任一源分支前停止并保留可诊断的检出失败信息。当被占用目标分支不等于远端快照时，同样沿用停止行为。

### 从已同步目标分支安全重试

如果源分支删除在当前工作树已切回目标分支后失败，下一次 cleanup（清理）只在当前 `HEAD`、本地目标分支和本次远端快照一致时接受该目标分支作为起点。远端目标已推进时仍停止，避免更新当前检出的分支或覆盖本地状态。

### 串行化共享目标分支的检出决定

同一 common Git dir（共享 Git 目录）与目标分支组合使用现有 PR Flow（拉取请求流程）锁目录中的短生命周期文件锁。锁只覆盖重新读取目标分支占用、更新本地目标引用、检出或降级为 detached HEAD（分离头）及一致性回读；不会包住 PR（拉取请求）查询、检查等待或源分支删除。

第二个 cleanup（清理）获得锁后必须重新读取工作树清单：第一个 cleanup（清理）已检出目标分支时，第二个按已同步占用路径保持 detached HEAD（分离头）。这样避免两个并发工作树同时检出同一目标分支，同时保留它们各自独立的流程锁。

### 以回读确认替代推断

成功路径在删除源分支后重新读取当前 `HEAD`、本地目标分支和本次远端快照。未被占用时三者必须一致，且完成详情标记 `baseCheckout.status: checked_out`；被占用时当前 `HEAD` 和本地目标分支仍必须等于快照，完成详情标记跳过原因。

这比只依赖命令返回码更能发现错误检出、过期引用或错误状态记录。

## Risks / Trade-offs

- [两个 cleanup（清理）同时检出同一目标分支] -> 用共享 Git（版本管理）目录中的短生命周期目标分支锁串行化检出决定；后获得锁的一方重新读取占用状态后降级。
- [目标分支在预检后被其他工作树占用] -> 重新读取工作树清单；已同步则保留 detached HEAD（分离头）完成清理，否则在删除源分支前停止。
- [目标分支检出失败导致工作树状态变化] -> 在删除任一源分支前完成检出与一致性回读；失败时不删除源分支。
- [其他工作树仍使用目标分支] -> 仅在其已等于远端快照时降级为 detached HEAD（分离头），绝不更新或检出该分支。
- [远端在流程中推进] -> 固定本次 fetch（拉取）快照；不额外拉取，调用方可在后续流程重新运行 cleanup（清理）。
