## Why

Cross Agent Review（跨代理审查）当前把所有差异无差别交给两个审查角色，Comet（双星工作流）产生的大量规格、设计、计划和流程文档因此成为主要噪音；任一角色超时或后续只发生机械变化时，又会触发整轮重审，放大成本和结果波动。同时，`mark-pass`（标记通过）把 Agent Guard（代理守卫）的证据路径、字段和写入职责耦合进了审查插件，与“主代理作出通过结论、守卫负责证据契约”的边界冲突。

## What Changes

- Cross Agent Review（跨代理审查）基于调用方声明和文件清单建立角色化审查范围：规格对齐角色关注权威上下文与相关实现，实施正确性角色优先关注实现、测试和配置；未分类文件默认完整审查，不按扩展名、文件大小或 Comet（双星工作流）目录硬编码排除。
- 新增持久化 `review-state.json`（审查状态文件），逐角色原子记录输入范围、尝试、完成、失败、超时和复用状态，使成功角色不会因另一角色失败而丢失。
- 新增只重试失败或超时角色的 `retry`（重试）入口，以及只接受 `checkbox-only`（仅复选框）和 `mapping-fields-only`（仅映射字段）两类声明式机械变化的跨提交 `revalidate`（重新校验）入口；任何不明确、重叠、重命名、复制、解析失败、规格或设计变化均退回真实审查。
- `revalidate`（重新校验）在满足复用规则时只生成绑定当前 `HEAD`（当前提交头）的新报告和状态，不自动作出通过结论，也不写通过证据。
- Agent Guard（代理守卫）新增通用 `record-evidence`（记录证据）入口：主代理完成语义判断后显式调用；入口从 `artifacts.yaml`（产物注册文件）解析画像、产物和安全路径，校验证据归属、当前提交头和干净工作区，并以原子方式写入现有 `guard-evidence/v1`（守卫证据第一版）格式。
- `record-evidence`（记录证据）注入保留的标准字段，调用方只提供业务字段；保留字段冲突、未知画像或产物、路径越界、提交头不一致、工作区不干净均拒绝写入。
- Cross Agent Review（跨代理审查）与 Planning Review（规划审查）都可由主代理通过同一个通用入口写入各自 Guard Profile（守卫画像）声明的通过证据；审查技能本身保持只输出审查事实。
- **BREAKING（不兼容变更）**：删除 Cross Agent Review（跨代理审查）的 `mark-pass`（标记通过）命令，以及其中所有 Agent Guard（代理守卫）画像、产物、路径和证据字段知识。
- 保持 Comet（双星工作流）原有阶段链、技能、脚本和状态推进不变；只更新外部 Guard Profile（守卫画像）集成契约。保持当前完整 `HEAD`（提交头）绑定、12 位短提交路径和现有门禁检查不变。

## Capabilities

### New Capabilities

无。

### Modified Capabilities

- `cross-agent-review`（跨代理审查）：增加角色化输入范围、可恢复状态、失败角色重试和机械变化复用，并移除通过证据写入职责。
- `agent-guard-core`（代理守卫核心）：允许主代理显式请求通用、机械、受契约约束的证据写入，同时继续禁止守卫自主作出通过结论。
- `agent-guard-plugin-runtime`（代理守卫插件运行时）：增加 `record-evidence`（记录证据）的解析、校验和原子写入契约，修正“只读不写”的过宽边界。
- `comet-agent-review-gate`（双星代理审查门禁）：把跨代理审查与规划审查的通过标记统一改为由主代理调用通用证据入口写入，保留现有门禁路径和字段检查。

## Impact

- 影响 `plugins/cross-agent-review`（跨代理审查插件）和 `plugins/agent-guard`（代理守卫插件）的命令、运行脚本、技能说明、测试与发布清单。
- 影响 Cross Agent Review（跨代理审查）、Agent Guard（代理守卫）和 Comet review gate（双星审查门禁）的现行 OpenSpec（开放规格）契约及对应规则文档。
- 不修改 Comet（双星工作流）的五阶段技能、状态文件、阶段脚本或推进逻辑；不新增依赖、数据库、配置框架或插件系统。
- 本变更只修改仓库源码和规格，不安装或同步用户级 Plugin（插件）或 Guard Profile（守卫画像）。
- 本次采用单一 change（变更），因为通用证据写入与旧 `mark-pass`（标记通过）删除构成一个原子迁移，并共享同一条端到端验收路径；拆分会产生临时双写或无人写状态。
