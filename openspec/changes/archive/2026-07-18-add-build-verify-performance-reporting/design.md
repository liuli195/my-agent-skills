## Context

Build and Verify（构建与验证）运行器已经为每个 verify check（验证检查项）记录单次耗时，并在 `--full`（完整验证）模式中运行全部检查且不读取缓存跳过检查。当前缺少完整验证墙钟总耗时、仓库可选预算、固定诊断报告和相应初始化契约。

本变更只扩展通用插件。现有规格中针对本仓库 Full（完整验证）的固定并行配置和 30 秒目标仍属于仓库自身验证约束，不作为插件默认值，也不写入通用模板。

## Goals / Non-Goals

**Goals:**

- 用一个可选正整数 `verify.fullBudgetSeconds`（完整验证预算秒数）表达目标仓库自己的总耗时预算。
- 完整运行所有验证检查后再判断是否超预算；性能结果不得改变功能验证退出状态。
- 超预算时自动输出性能警告并记录固定 JSON（结构化数据）报告。
- 未超预算时允许用户通过 `--performance-report`（性能报告）显式记录同一格式报告。
- 保持插件不安装依赖、不识别业务测试框架、不修改 CI（持续集成）的现有边界。

**Non-Goals:**

- 不在达到预算时提前终止检查。
- 不增加逐项预算、历史趋势、版本基线、统计采样、自动调优或二分定位。
- 不修改本仓库 `.build-and-verify/config.json`、业务检查命令或仓库专用 30 秒目标。
- 不保存多份历史报告。

## Decisions

### 1. 复用现有 verify（验证）命令

在现有 `verify`（验证）参数中增加布尔参数 `--performance-report`（性能报告），且只允许与 `--full`（完整验证）同时使用。不新增顶层 performance（性能）命令或独立编排模块，因为总计时和报告都能在现有运行器结果汇总点完成。

替代方案是新增 `performance diagnose`（性能诊断）子命令；它会重复配置读取、检查调度和结果传播，因此不采用。

### 2. 预算是完成后警告，不是超时

`verify.fullBudgetSeconds`（完整验证预算秒数）只接受正整数。计时使用 Python（Python 语言）标准库 `time.monotonic()`，从开始调度完整检查到所有检查结果返回为止；配置读取、参数解析和报告写入不计入预算。公开总耗时先统一保留两位小数，再用该值判断是否超预算并生成终端和报告内容，避免同一次运行出现互相矛盾的公开数值。

运行器始终完成全部检查，再比较总耗时与预算。超预算设置报告字段并输出 `performance-warning`（性能警告），但最终退出状态仍只由功能检查、配置错误和参数错误决定。

现有并行中断路径可能只返回部分检查结果。若结果数与所选检查数不一致，本次运行不得判断预算、输出性能警告或写入报告；功能验证仍按现有失败路径返回。

### 3. 条件生成一个固定报告

报告触发条件固定为：

| 超预算 | 显式报告参数 | 是否写报告 |
|---|---|---|
| 否 | 否 | 否 |
| 否 | 是 | 是 |
| 是 | 否 | 是 |
| 是 | 是 | 是 |

报告固定覆盖 `.build-and-verify/runs/performance-report.json`。写入使用同目录临时文件加原子替换，避免留下半份 JSON（结构化数据）。报告目录不存在时创建；写入失败只输出 `performance-report-warning`（性能报告警告），不改变原验证退出状态。

当一次完整验证既未超预算也未请求报告时，不创建、不覆盖也不删除固定报告。固定报告只表示最后一次实际生成的性能诊断，读取者使用报告生成时间识别其对应运行。

### 4. 固定且最小的报告契约

报告使用 `schemaVersion: 1`（结构版本 1），固定包含：

- `runtimeVersion`（运行时版本）。
- `generatedAt`（报告生成时间，UTC（协调世界时））。
- `totalSeconds`（总秒数）。
- `budgetSeconds`（预算秒数），未配置时为 `null`。
- `overBudget`（是否超预算），未配置时为 `null`。
- `verificationStatus`（验证状态），取 `passed` 或 `failed`。
- `checks`（检查项），按配置顺序记录标识、状态和耗时。

现有 `CheckResult`（检查结果）继续作为单项数据来源；调度层保留并返回结果列表供最终汇总，不解析业务命令输出。

### 5. 初始化能力保持可选

通用配置模板不写入默认预算。Build and Verify Init（构建与验证初始化）问卷只在用户选择启用完整验证预算时，将确认后的正整数写入 `verify.fullBudgetSeconds`，并在最终摘要和写后校验中展示。

## Risks / Trade-offs

- **固定报告每次覆盖，无法观察趋势** → 当前只承诺本次诊断；需要趋势时再设计独立历史能力。
- **墙钟时间受机器负载影响** → 报告明确是警告而非阻断，不把单次波动当成功能失败。
- **旧报告可能被误认为当前结果** → 报告固定记录 `generatedAt`（生成时间），且未触发报告的运行不修改该文件。
- **报告写入失败导致诊断缺失** → 明确输出警告，但保留原验证结果，避免诊断设施阻断验证。

## Migration Plan

1. 发布更新后的插件源码和双平台 manifest（清单）版本。
2. 目标仓库显式运行 `update-runtime`（更新运行时）后获得新能力；插件不自动修改仓库运行时。
3. 已有配置无需迁移；未声明 `verify.fullBudgetSeconds` 时行为保持不变。
4. 回滚到旧运行时会忽略新命令参数并不支持新配置，因此回滚前应移除相应调用；不会影响既有检查定义。
