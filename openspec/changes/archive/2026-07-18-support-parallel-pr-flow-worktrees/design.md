## Context

PR Flow（拉取请求流程）目前以调用目录和单一 `.pr-flow/last-status.json` 为中心，没有识别 linked worktree（关联工作树）的运行身份，也没有防止同一工作树上的修改命令重入。complete（完整流程）和 tweak（小改）在合并前没有把 required checks（必需检查）、review gate（审查门禁）与同一组源/目标提交绑定；cleanup（清理）还会切换本地目标分支，因此会与其他工作树的分支占用冲突。

本改动必须兼容现有单工作树使用方式，复用 `.pr-flow/runs/`，只依赖 Python（编程语言）标准库、Git（版本控制）和 GitHub CLI（GitHub 命令行工具）。

## Goals / Non-Goals

**Goals:**

- 每个工作树独立记录和恢复 complete（完整流程）、tweak（小改）、cleanup（清理）与 hotfix（热修复）状态。
- 同一工作树与分支上的修改命令互斥，不同工作树互不阻塞；diagnose（诊断）始终只读可用。
- 每次分发、门禁、合并和 hotfix（热修复）推送都基于当时最新的远端目标提交。
- cleanup（清理）不切换本地目标分支，并能安全处理显式关联工作树删除。
- 保持现有命令、配置和单工作树流程兼容。

**Non-Goals:**

- 自动 rebase（变基）、merge（合并）或解决业务冲突。
- 跨工作树修改文件、自动清理其他工作树，或提供仓库级全局锁。
- 新增共享 Runtime（运行时）、第三方依赖、配置开关或长期保留已删除工作树的运行状态。

## Decisions

### 1. 运行上下文完整记录，每个工作树只使用一把修改锁

每次调用先解析 common Git dir（共享 Git 目录）、规范化工作树路径、当前源分支、目标分支、流程类型和 PR（拉取请求，允许为空），并把源/目标提交写入状态。运行状态由当前源分支识别；PR（拉取请求）在创建或发现后更新到同一份状态中。

每个工作树的所有修改命令共用一把固定的非阻塞独占锁。锁位于 `<common-git-dir>/pr-flow-locks/<worktree-key>.lock`，`worktree-key` 直接使用 `hashlib.sha256` 根据规范化工作树路径生成。complete（完整流程）进入内部 cleanup（清理）时复用同一锁；不同工作树使用各自的锁，可以并行运行。

### 2. 复用工作树本地 `.pr-flow/runs/`

运行状态写入目标工作树的 `.pr-flow/runs/<branch-key>.json`；`branch-key` 直接使用 Python（编程语言）标准库 `hashlib.sha256` 根据源分支生成。现有 `write_status` 负责更新该状态和 `.pr-flow/last-status.json`，不新增第二套状态输出层。

锁只用于协调，不复制运行状态。它位于工作树外，因此从其他工作树显式删除目标工作树时，不会因打开的锁文件阻止 Windows（微软操作系统）删除目录。

不再向 common Git dir（共享 Git 目录）复制第二份规范状态。显式删除关联工作树时，该工作树的运行记录随之删除，这是一次性工作树的预期生命周期。

锁使用 Python（编程语言）标准库提供的操作系统文件锁：Windows（微软操作系统）使用 `msvcrt`，其他平台使用 `fcntl`。拿锁失败时立即输出 `DISPATCH_REQUIRED` 和 `flow_locked`，不等待，也不调用 `write_status` 改写该工作树的运行状态或 `.pr-flow/last-status.json`。

diagnose（诊断）只读取锁与状态并输出结果。若发现修改命令持锁，它不得覆盖该流程状态或 `.pr-flow/last-status.json`；若无持锁流程，则保留现有诊断状态写入行为。锁冲突输出复用现有下一步命令和 actor（执行者）信息生成能力。

### 3. 从 Git 的空字符分隔清单识别工作树

工作树发现使用 `git worktree list --porcelain -z`，避免空格、中文和换行转义造成路径误判。所有路径先解析为绝对规范路径，再用于身份、占用和删除判断。源分支占用检查排除当前目标工作树，但任何其他工作树占用都会在删除前停止。

### 4. 分发前验证最新远端目标基线

现有 hotfix（热修复）目标提交读取逻辑收敛为一个远端提交快照函数。diagnose（诊断）在给出分发动作前、complete（完整流程）和 tweak（小改）在自动推送或创建、同步、合并 PR（拉取请求）等远端修改前，复用该函数读取最新远端目标提交；只读的 PR（拉取请求）查询不受限制。只有最新远端目标提交是当前源提交的祖先时才继续；否则统一输出 `base_outdated` 停止状态，不额外判断是否冲突，也不自动改写源分支。

这比自动变基或合并更可预测，也让两个并行 PR（拉取请求）在目标分支推进后重新接受验证。

### 5. 门禁绑定同一组源/目标提交

等待门禁前读取 PR（拉取请求）的源/目标提交快照。现有 `pr_checks` 和 `wait_for_checks` 改为读取 GitHub CLI（GitHub 命令行工具）的 required checks（必需检查）；结果必须非空，且每项都属于当前源提交并成功，缺失、等待、失败或过期都停止。等待状态沿用 `checks_pending`，其他阻断状态沿用 `checks_or_review_blocking`，不新增原因代码、检查模型或等待器。

review gate（审查门禁）完成后，在调用 `gh pr merge`（合并拉取请求命令）前最后一次读取源/目标提交。任一提交变化都会废弃本轮检查、审查和可合并结果，并要求重新运行。最终合并继续使用 `--match-head-commit` 固定源提交；目标提交以调用前复核和 GitHub（代码托管平台）现有合并门禁为准，不承诺平台不支持的目标提交原子绑定。

tweak（小改）复用完整生命周期，只跳过 review gate（审查门禁），不跳过基线、必需检查或提交复核。

### 6. hotfix 在推送前复核远端目标

hotfix（热修复）使用同一个远端提交快照函数，在验证前后分别读取并比较目标提交。目标已推进时停止，要求同步后重新验证；不复用旧验证结果，也不进入 PR（拉取请求）门禁。

### 7. cleanup 留在最新远端目标提交

cleanup（清理）先完成全部只读预检：PR（拉取请求）已合并、工作树干净、源/目标分支匹配、源分支未被其他工作树占用、远端目标提交可用。预检通过后，将当前目标工作树切到最新远端目标提交的 detached HEAD（分离头），再删除远端和本地源分支，并输出最终提交与分离头状态。

默认保留工作树。只有显式 `--remove-worktree`（删除工作树参数）时才尝试删除；共享删除路径只接受 `git worktree list`（工作树清单）中登记的非主工作树和干净工作区，并且始终使用原生 `git worktree remove` 且不传 `--force`（强制）。cleanup（清理）、complete（完整流程）和 tweak（小改）只有完成已合并 PR（拉取请求）的清理预检后才进入共享删除路径；hotfix（热修复）只有推送回读成功且当前提交等于最新远端目标提交时才进入，不查询 PR（拉取请求）。不新增独立的“流程归属”状态或匹配系统。从待删除工作树内部运行时，命令只完成对应流程的安全收尾并返回一条需从外部执行的删除命令；从外部通过 `--project`（项目路径参数）运行时，删除后回读工作树清单。

## Risks / Trade-offs

- [修改命令异常退出后锁文件仍存在] → 操作系统锁随进程释放；锁文件存在本身不代表占用，诊断以实际加锁结果为准。
- [目标分支频繁推进导致流程重复停止] → 停止状态提供同步和重试命令；不复用旧门禁结果。
- [工作树删除导致本地运行记录消失] → 这是显式删除的一部分；合并与远端 PR（拉取请求）记录仍保留，不增加第二份状态源。
- [清理在删除远端分支后、本地删除前失败] → 重试时读取实时 Git（版本控制）状态，将已不存在的远端或本地源分支视为已完成，不增加步骤检查点。

## Migration Plan

无配置迁移；回滚时恢复脚本与 Skill（技能）说明即可。
