## 背景

`find_pr()`（PR 查询函数）直接返回 `gh pr view`（PR 查看命令）的结果，未区分 PR（拉取请求）状态。`complete`（收尾）和 `tweak`（小改）因此可能将旧批次的源提交用于新批次的基线校验。

## 目标与非目标

**目标：**

- 仅将 `OPEN`（未合并）PR（拉取请求）作为活动 PR（拉取请求）。
- 终态 PR（拉取请求）后续批次使用当前 `HEAD`（当前提交）继续既有创建流程。

**非目标：**

- 不自动变基、解决冲突或修改历史 PR（拉取请求）。
- 不改变活动 PR（拉取请求）的检查、审查或合并逻辑。

## 决策

- 在既有 `find_pr()`（PR 查询函数）集中筛选状态；所有生命周期调用方都会获得同一语义，避免在各调用方重复判断。
- 终态状态返回“无活动 PR”，复用已有的“未找到 PR（拉取请求）”创建路径和当前 `HEAD`（当前提交）基线校验。
- 测试使用现有命令桩（测试替身），覆盖已合并、已关闭及活动三种状态，不新增测试框架或辅助层。

## 风险与取舍

- [GitHub（代码托管平台）返回未知状态] → 仅 `OPEN`（未合并）被视为活动，未知状态也不会误用旧源提交；创建 PR（拉取请求）若被远端拒绝，保持现有停止状态。
- [状态筛选影响恢复路径] → 回归覆盖活动 `OPEN`（未合并）PR（拉取请求）的现有合并路径。
