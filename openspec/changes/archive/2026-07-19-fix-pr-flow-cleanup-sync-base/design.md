## Context

cleanup（清理）原本会检出并快进本地 base branch（目标分支）。并行 worktree（工作树）支持为了避免切换被其他工作树占用的目标分支，改为把当前工作树留在最新远端目标提交的 detached HEAD（分离头），但同时遗漏了本地目标分支引用的同步。

分离当前工作树和同步本地分支引用是两个独立动作，可以同时满足。约束是不得覆盖本地独有提交，也不得使正在其他工作树中检出的目标分支与其文件状态失配。

## Goals / Non-Goals

**Goals:**

- cleanup（清理）成功后，当前 `HEAD`、本地目标分支和已读取的最新远端目标提交一致。
- 保留 detached HEAD（分离头）和多工作树安全边界。
- 失败可重试，且在删除源分支前完成同步安全检查。

**Non-Goals:**

- 不自动变基、合并或丢弃本地目标分支的独有提交。
- 不切换或修改其他工作树的检出状态。
- 不新增锁、配置、命令或第三方依赖。

## Decisions

### 1. 分离工作树与同步分支引用分别处理

继续使用现有远端快照得到 `base_oid`（目标提交）。当前工作树定位到该提交的 detached HEAD（分离头）；本地目标分支引用独立快进到同一提交，不需要检出目标分支。

这保留了多工作树方案避免分支切换冲突的目的，同时恢复本地与远端目标分支一致的契约。

### 2. 只允许安全快进

本地目标分支不存在时可在 `base_oid`（目标提交）创建；已存在时必须等于该提交或是其祖先。若本地目标分支包含远端没有的提交，cleanup（清理）停止，不覆盖数据。

如果其他工作树正在检出本地目标分支，只有该分支已经等于 `base_oid`（目标提交）时才能继续；否则停止并报告占用工作树，由该工作树先同步。

### 3. 完成状态包含一致性不变量

只有当前 `HEAD`、本地目标分支和本轮远端目标快照都等于 `base_oid`（目标提交），才能写入 `cleanup_complete`（清理完成）。重试路径执行同一检查，不新增恢复状态机。

## Risks / Trade-offs

- [本地目标分支包含独有提交] → 停止并保留提交，由用户决定如何整合。
- [其他工作树检出落后的目标分支] → 停止并报告路径，避免静默改写该工作树的文件状态。
- [远端在快照后再次推进] → 本轮仍以已固定快照完成；后续流程会重新读取最新远端基线。
